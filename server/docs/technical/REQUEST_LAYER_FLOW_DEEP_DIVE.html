Request Layer Flow Deep Dive

מטרה: לתעד בצורה עמוקה את הזרימה של בקשה דרך כל שכבות השרת, כולל חוזה קלט (מה חובה ומה רשות), לוגיקה פנימית, תלויות, שגיאות ותוצר סופי.

Endpoint 01 (לפי סדר לוגי): הרשמת משתמש

שם תרחיש: משתמש חדש נרשם למערכת

Method: POST
Path: /api/auth/register
Full URL (שרת מקומי): http://localhost:5000/api/auth/register
Auth: לא נדרש
Rate limit: כן, authRateLimiter

1) חוזה בקשה מפורט (Input Contract)

1.1 Body JSON

שדה סוג חובה/רשות חוקי ולידציה הערות עומק
name string חובה מינימום 2 תווים נשמר ב-User.name
email string חובה חייב פורמט אימייל תקין מנורמל ל-lowercase לפני שמירה
password string חובה מינימום 6 תווים יעבור hashing ב-pre-save middleware

1.2 שדות שנשלחים אך לא נצרכים ב-Register הנוכחי

שדה מצב בפועל השפעה
confirmPassword לא מוגדר בסכימת registerSchema לא נלקח ל-service (מתעלם)
phone לא מוגדר בסכימת registerSchema לא נלקח ל-service (מתעלם)
כל שדה נוסף לא מוגדר בסכימת registerSchema לא נלקח ל-service (מתעלם)

הערת עומק: בזרימה הנוכחית, נקודת הכניסה היא parse על registerSchema ולכן רק name/email/password ממשיכים הלאה.

1.3 Headers/Params/Query

רכיב חובה/רשות ערך נדרש
Authorization לא נדרש אין
URL params אין אין
Query params אין אין
Content-Type מומלץ application/json

2) שכבות הזרימה (Layer by Layer)

2.1 שכבת App גלובלית (לפני Router)

סדר שכבה קובץ מה היא עושה קלט פלט/השפעה תקלות אפשריות
1 helmet src/app.ts מוסיף Security Headers HTTP Request Response headers קשיחים יותר לרוב לא נכשל בתרחיש תקין
2 corsConfig src/app.ts בודק הרשאות CORS Origin/Method/Header מאשר/חוסם cross-origin חסימת CORS בדפדפן
3 express.json src/app.ts פרסור JSON body Raw body req.body אובייקט JSON לא תקין או גדול מדי
4 express.urlencoded src/app.ts תמיכה ב-urlencoded Form body req.body לא רלוונטי ל-JSON רגיל
5 requestIdMiddleware src/app.ts יוצר מזהה בקשה req X-Request-ID + הקשר לוגים ללא השפעה עסקית
6 auditLoggingMiddleware src/app.ts מעשיר context audit IP, UA נתוני מעקב ללא השפעה עסקית
7 requestLoggerMiddleware src/app.ts לוג בקשה/תגובה req/res תיעוד מלא בלוגים ללא השפעה עסקית
8 metricsMiddleware src/app.ts מדדי Prometheus req/res counters/histograms ללא השפעה עסקית

סדר שכבה קובץ מה היא עושה קלט פלט/השפעה תקלות אפשריות

2.3 שכבת Middleware ייעודית ל-endpoint

סדר שכבה קובץ מה היא עושה קלט פלט/השפעה תקלות אפשריות
11 authRateLimiter src/middlewares/rate-limiter.middleware.ts מגביל ניסיונות על auth endpoints req.ip (או userId אם קיים) מאפשר/חוסם המשך 429 RATE_LIMITED

מדיניות בתרחיש זה: אם חרגנו מהחלון המותר, מוחזר 429 עם success=false וקוד RATE_LIMITED.

2.4 שכבת Controller

סדר שכבה קובץ מה היא עושה קלט פלט/השפעה תקלות אפשריות
12 asyncHandler wrapper src/utils/asyncHandler.ts עוטף שגיאות async ומעביר ל-error middleware Promise errors next(error) במקום crash ללא wrapper עלול להיות unhandled rejection
13 AuthController.register src/controllers/auth.controller.ts parse ל-body ואז קריאה ל-service req.body response 201 עם data Zod validation error 400

פירוק Controller.register:

  1. registerSchema.parse על req.body
  2. AuthService.register עם נתונים שעברו ולידציה
  3. מחזיר 201 עם success=true, data=result, message

2.5 שכבת Validation

סדר שכבה קובץ מה היא עושה חוקים כישלון
14 registerSchema (Zod) src/validators/auth.validator.ts מאמת name/email/password name min 2, email format, password min 6 ZodError -> 400 VALIDATION_ERROR

2.6 שכבת Service (Business Logic)

סדר שכבה קובץ מה היא עושה קלט פלט/השפעה תקלות אפשריות
15 AuthService.register src/services/auth.service.ts לוגיקת הרשמה מלאה CreateUserInput user + token + refreshToken 409 CONFLICT אם אימייל קיים

פירוק לוגי בתוך AuthService.register:

  1. חיפוש משתמש קיים לפי email lowercase
  2. אם קיים: זריקת ApiError עם 409 ו-code CONFLICT
  3. יצירת משתמש חדש ב-UserModel.create
  4. הנפקת access token עם tokenVersion נוכחי
  5. הנפקת refresh token עם tokenVersion נוכחי
  6. עדכון lastLogin ושמירה
  7. החזרת sanitizeUser + tokens

2.7 שכבת Model + MongoDB

סדר שכבה קובץ מה היא עושה קלט פלט/השפעה תקלות אפשריות

3) שכבת Error Handling חוצה מערכת

מקור שגיאה דוגמה תרגום ל-HTTP מבנה תגובה
ApiError duplicate email statusCode מהשגיאה (למשל 409) success=false, message, errors, code
ZodError email לא תקין 400 success=false, message=Validation failed, errors, code=VALIDATION_ERROR
שגיאה כללית DB/network runtime 500 (או status קיים) success=false, code=INTERNAL_ERROR

error middleware: src/middlewares/error.middleware.ts

4) תגובות צפויות (Response Contract)

4.1 Success 201

שדה תיאור
success true
data.user פרטי משתמש מסוננים (ללא password)
data.token JWT access token
data.refreshToken JWT refresh token
message User registered successfully

4.2 Error 400

סיבה דוגמה
שדה חסר/לא תקין email חסר, name קצר מדי, password קצר מדי

4.3 Error 409

סיבה דוגמה
אימייל כבר קיים אותו email נרשם כבר במערכת

4.4 Error 429

סיבה דוגמה
קצב בקשות גבוה מדי יותר מדי ניסיונות הרשמה בפרק זמן מוגדר

5 POST /api/auth/register Validation registerSchema name>=2, email תקין, password>=6 400 או ממשיך
6 POST /api/auth/register Service AuthService.register email לא קיים 409 או יצירה
7 POST /api/auth/register Model/DB UserModel + pre-save hash password נשמר hashed יצירת משתמש + timestamps
8 POST /api/auth/register Response Controller output success payload 201 עם user+token+refreshToken
